# Skill from Notebook

从学习材料（文档、文章、视频）或高质量示例（博客、设计、代码）中提取可操作的方法论，生成可复用的 Skills。

**核心哲学**：NotebookLM 帮你理解。这个 Skill 帮你**执行**。

## 使用场景

当用户想把知识转化为可执行技能时：

- "我刚读了篇关于代码评审的文章，帮我从中创建一个 skill"
- "这篇技术博客很棒，提取写作方法论"
- "把这个 PDF 指南变成我可以复用的 skill"
- "从这个示例学习，创建一个能产出类似输出的 skill"

## 支持的输入类型

| 类型 | 如何处理 |
|------|----------|
| 本地文件 | PDF、Word、Markdown — 直接读取 |
| 网页 URL | WebFetch 提取内容 |
| YouTube | 使用工具获取字幕 |
| 示例/输出 | 逆向工程方法论 |

## 步骤 0: 识别输入类型

**关键第一步** — 确定使用哪个处理路径：

```
用户输入
    │
    ├─ 有教学意图？("如何"、"步骤"、"指南")
    │   └─ 是 → 路径 A: 方法论文档
    │
    ├─ 是完成的作品？(文章、设计、代码、提案)
    │   └─ 是 → 路径 B: 示例（逆向工程）
    │
    └─ 都不是？→ 告诉用户这个内容不适合
```

**路径 A 指示**（方法论文档）：
- 包含"如何"、"步骤"、"方法"、"指南"等词
- 有编号列表或步骤序列
- 有教学意图
- 描述"做什么"

**路径 B 指示**（示例/输出）：
- 是完整的作品/产物
- 没有教学意图
- 是"事物本身"而非"如何制作事物"
- 例如：写得好的博客文章、精致的提案、代码项目

## 路径 A: 从方法论文档提取

### A1: 验证文档适用性

检查文档是否适合生成 Skill（至少满足 2 个）：

- [ ] 有明确目标/结果
- [ ] 有可重复的步骤/流程
- [ ] 有质量标准
- [ ] 有场景/上下文描述

**如果不合适**：诚实告知用户并解释原因。

### A2: 识别 Skill 类型

| 类型 | 特征 | 示例 |
|------|------|------|
| **How-to** | 清晰步骤，输入→输出 | 部署 Docker、配置 CI/CD |
| **Decision** | 条件、权衡、选择 | 选择数据库、选择框架 |
| **Framework** | 心智模型、分析维度 | SWOT、5W1H、第一性原理 |
| **Checklist** | 验证清单、通过/失败标准 | 代码审查清单、上线清单 |

### A3: 按类型提取结构

**How-to：**
- 前置条件
- 步骤序列（每步预期输出）
- 最终预期结果
- 常见错误

**Decision：**
- 决策因素
- 选项及优缺点
- 决策树/流程图
- 推荐默认选择

**Framework：**
- 核心概念
- 分析维度
- 应用方法
- 局限性

**Checklist：**
- 检查项目及标准
- 优先级
- 常见遗漏项

### A4: 生成 Skill

使用模板：

```markdown
## 适用场景
[何时使用此 Skill]

## 前置条件
- [开始前需要什么]

## 步骤
1. [步骤 1] - [预期结果]
2. [步骤 2] - [预期结果]
...

## 质量检查点
- [ ] [检查点 1]
- [ ] [检查点 2]

## 常见陷阱
- [陷阱 1]: [如何避免]

## 来源
- 文档: [名称/URL]
- 提取时间: [时间戳]
```

## 路径 B: 从示例逆向工程

当输入是完成的作品（不是教程），逆向工程方法论。

### B1: 识别输出类型

这是什么类型的产物？
- 技术博客文章
- 产品提案/PRD
- 学术论文
- 代码架构
- 设计文档
- 其他：[指定]

### B2: 分析结构

分解示例：

```
结构分析：
├── [部分 1]: [功能] - [占比 %]
├── [部分 2]: [功能] - [占比 %]
├── [部分 3]: [功能] - [占比 %]
└── [部分 N]: [功能] - [占比 %]
```

需要回答的问题：
- 它有多少部分？
- 每个部分的功能是什么？
- 顺序和比例如何？

### B3: 提取质量特征

是什么让这个示例**好**？

| 维度 | 问题 |
|------|------|
| 结构 | 内容如何组织？ |
| 风格 | 语气、措辞、表达？ |
| 技术 | 什么方法让它有效？ |
| 逻辑 | 信息如何流动？ |
| 细节 | 小但重要的点？ |

### B4: 逆向工程流程

推断：创建这个输出，需要什么步骤？

```markdown
## 推断的生产步骤
1. [步骤 1]: [做什么] - [关键点]
2. [步骤 2]: [做什么] - [关键点]
...

## 关键决策
- [决策 1]: [选项] - [本例选择 X 因为...]

## 可复用技术
- [技术 1]: [如何应用]
- [技术 2]: [如何应用]
```

### B5: 生成 Skill

使用逆向工程的 Skill 模板：

```markdown
## 输出类型
[这个产生什么类型的产物]

## 适用场景
[何时创建这种输出]

## 结构模板
1. [部分 1]: [功能] - [~X%]
2. [部分 2]: [功能] - [~X%]
...

## 质量特征（从示例学习）
- [特征 1]: [如何体现]
- [特征 2]: [如何体现]

## 生产步骤
1. [步骤 1]: [做什么] - [技巧]
2. [步骤 2]: [做什么] - [技巧]
...

## 检查清单
- [ ] [检查项 1]
- [ ] [检查项 2]

## 参考示例
- 来源: [名称/URL]
- 分析时间: [时间戳]
```

## 示例：路径 A（方法论文档）

**用户**: "从这篇关于写好 commit messages 的文章中提取 skill"

**流程**：
1. 阅读文章
2. 识别：这是 **How-to** 类型（有步骤、教学意图）
3. 提取：
   - 目标：写出清晰有用的 commit messages
   - 步骤：使用 conventional 格式、分隔 subject/body 等
   - 质量标准：subject < 50 字、祈使语气等
4. 生成带步骤和检查清单的 skill

---

## 示例：路径 B（逆向工程）

**用户**: "这篇技术博客很棒。学习它并创建一个写类似文章的 skill"

**流程**：
1. 识别：这是**示例**（完成作品，没有教学意图）
2. 分析结构：
   ```
   ├── Hook: 真实的痛点（2-3 句）
   ├── Problem: 3 句核心问题
   ├── Solution: 先结论再细节
   ├── Code: 每段 < 20 行，带注释
   ├── Pitfalls: 3 个常见错误
   └── Summary: 一句话总结
   ```
3. 提取质量特征：
   - 标题 = 具体技术 + 解决的问题
   - 每段一个想法
   - 代码:文字比例 ~40:60
   - 个人轶事增加可信度
4. 逆向工程步骤：
   - 从你解决的真实问题开始
   - 先写解决方案，再写铺垫
   - 逐步添加代码示例
   - 等等
5. 生成 Skill："如何写技术博客文章"

## 高级：多示例学习

当用户提供多个同类型示例时：

```
示例 A ──┐
示例 B ──┼──> 提取共性 ──> 核心方法论
示例 C ──┘           │
                    ▼
                  分析差异 ──> 风格变体 / 可选技术
```

这会产生更健壮、更通用的 Skill。

---

## 重要说明

1. **始终先验证** — 不是所有内容都适合提取 Skill
2. **尽早识别路径** — 方法论文档 vs 示例需要不同方法
3. **保持具体** — 模糊的 Skill 没用；包含具体步骤和标准
4. **保留来源** — 始终注明知识来源
5. **不确定时询问** — 如果不确定用户意图，先问再继续
6. **质量优先** — 花时间真正理解内容

## 这个 Skill 不是

- 不是摘要器（那是 NotebookLM 的工作）
- 不是文档转换器
- 是提取**可操作方法论**——可以重复执行的步骤
