# 代码美学规则参考

*天才思维 + 美学大师 联合分析产出 — 道法术器四层框架*

---

## 道（第一性原理）

| 原理 | 说明 |
|------|------|
| **认知负荷极限** | 人类工作记忆 4±1 个信息块 |
| **模式识别本能** | 大脑渴望预测一致性 |
| **时间感知扭曲** | 调试 10分钟 > 编码 1小时 |
| **chunking 机制** | 信息组块化是专家与新手差异 |
| **信任建立需求** | 阅读者需快速建立正确性信心 |

---

## 法（方法论框架）

| 法则 | 要义 |
|------|------|
| 单一职责 | 一个单元只做一件事 |
| 开闭原则 | 对扩展开放，对修改关闭 |
| 最小惊讶 | 符合阅读者合理预期 |
| 局部性原理 | 相关概念物理邻近 |
| 渐进披露 | 复杂度分层隐藏 |
| 对称与一致 | 相似问题相似解法 |

---

## 术（执行技法）

### 命名术
```
✓ 动词+名词：get_user_by_id
✓ 布尔问句式：is_valid, has_permission
✓ 领域术语准确
✗ 单字母（循环除外）
✗ 模糊命名：processData
```

### 格式化
```
✓ 垂直对齐揭示关系
✓ 空行分隔逻辑段
✓ 缩进表达层级
✓ 行宽限制 80-120
```

### 结构化
```
✓ 卫语句提前返回
✓ 策略模式替代条件堆砌
✓ 管道替代嵌套
✓ 纯函数隔离副作用
```

### 分解术
| 指标 | 阈值 |
|------|------|
| 函数 | < 20 行 |
| 参数 | < 4 个 |
| 嵌套 | < 3 层 |
| 圈复杂度 | < 10 |

### 注释术
```
✓ 解释"为什么"非"是什么"
✓ TODO 带上下文
✓ API 文档即代码
✗ 删除注释掉的代码
```

## 代码美学 vs 视觉美学映射

| 视觉美学 | 代码美学 | 实例 |
|----------|---------|------|
| 格式塔-邻近性 | 相关代码物理邻近 | 变量声明紧挨使用处 |
| 格式塔-相似性 | 命名与结构一致 | getX/setX/isX 模式 |
| 格式塔-连续性 | 阅读动线流畅 | 早返回避免嵌套 |
| 格式塔-闭合性 | 块结构完整 | 括号对齐、折叠自明 |
| 色彩-情绪 | 语法高亮 | 关键字/变量/常量色彩区分 |
| 形式-比例 | 代码体积平衡 | 函数<20行、文件<400行 |
| 符号-语义 | 命名即文档 | filterActiveUsers |
| 时空-叙事 | 阅读节奏 | 抽象→具体排列 |
| 情感-心流 | 编码流畅度 | 智能补全、即时反馈 |
| 伦理-责任 | 代码可维护性 | 无技术债务、可测试 |

---

## 一句话总结

> 代码美学的**道**是认知效率，**法**是简洁与一致，**术**是命名与结构，**器**是自动化工具链。
> 好代码不是「写得巧」，而是「读得顺」——让未来的自己/他人，以最低认知成本建立信任与理解。
