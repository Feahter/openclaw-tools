# Skill from GitHub

当用户想完成某任务时，搜索高质量项目来学习，然后创建 Skill。

## 使用场景

当用户描述任务，你想找到现有工具/项目来学习时：
- "我想把 markdown 转为 PDF"
- "帮我分析客户评论的情感"
- "我需要从代码生成 API 文档"

## 工作流程

### 步骤 1: 理解用户意图

明确用户想要实现什么：
- 输入是什么？
- 期望输出是什么？
- 有没有限制（语言、框架等）？

### 步骤 2: 搜索 GitHub

搜索解决这个问题：

```
{任务关键词} language:{首选语言} stars:>100 sort:stars
```

**搜索技巧：**
- 先广泛，再缩窄
- 尝试不同的关键词组合
- 如果相关，包含 "cli"、"tool"、"library"

**质量过滤器（必须满足所有）：**
- Stars > 100（社区验证）
- 12 个月内更新（活跃维护）
- 有清晰的 README 文档
- 有实际代码（不只是 awesome-list）

### 步骤 3: 向用户展示选项

展示前 3-5 个候选：

```markdown
## 找到 X 个可能帮助的项目

### 选项 1: [project-name](github-url)
- Stars: xxx | 最后更新: xxx
- 功能：一句话描述
- 为什么好：特定优势

### 选项 2: ...

哪一个我应该深入了解？或者换个搜索方式？
```

**等待用户确认后再继续。**

### 步骤 4: 深入了解选中的项目

用户选择项目后，全面理解：

1. **读 README** — 理解目的、功能、使用方法
2. **读核心源码** — 理解工作原理
3. **查看示例** — 了解真实使用模式
4. **注意依赖** — 它依赖什么
5. **识别关键概念** — 其背后的心智模型

提取：
- 核心算法/方法
- 输入/输出格式
- 错误处理模式
- 代码中编码的最佳实践

### 步骤 5: 总结理解

向用户展示学到的内容：

```markdown
## 理解 [project-name]

### 核心方法
它如何解决问题...

### 关键技术
1. ...
2. ...

### 局限
- ...

### 我将在 Skill 中编码什么
- ...

这看起来对吗？应该继续创建 Skill 吗？
```

**等待用户确认。**

### 步骤 6: 创建 Skill

生成 Skill 时包含：

1. **明确目的** — Skill 做什么
2. **方法论** — 从项目中学到的方法
3. **逐步流程** — 具体指令
4. **质量标准** — 好的输出什么样
5. **常见陷阱** — 要避免什么（从项目 issues/docs 中学到）

**重要：** Skill 应该编码项目的*知识*，而不只是包装工具。Skill 应该即使没有原始项目安装也能工作。

## 示例

**用户**: "我想写出更好的 commit messages"

**步骤 2 - 搜索**：
```
commit message convention stars:>100 sort:stars
```

**步骤 3 - 展示**：
```
## 找到 4 个项目

### 选项 1: conventional-commits/conventionalcommits.org (4.2k ⭐)
- 人类和机器可读的 commit messages 规范
- 为什么：行业标准，被 Angular、Vue 等使用

### 选项 2: commitizen/cz-cli (15k ⭐)
- 创建 conventional commits 的 CLI 工具
- 为什么：久经考验，很多适配器

### 选项 3: carloscuesta/gitmoji (14k ⭐)
- Commit messages 的 emoji 指南
- 为什么：视觉化，有趣

哪一个你感兴趣？
```

**用户**: "选择 conventional commits"

**步骤 4 - 深入**：
- 阅读规范
- 理解类型（feat、fix、docs 等）
- 学习 scopes、breaking changes
- 研究真实案例

**步骤 5 - 总结**：
```
## 理解 Conventional Commits

### 核心方法
结构化格式：<type>(<scope>): <description>

### 关键类型
- feat: 新功能
- fix: 修复 bug
- docs: 文档
- ...

### 学到的好处
- 自动生成 changelog
- 语义化版本
- 清晰的历史

准备好基于此创建 commit message skill 了吗？
```

**步骤 6 - 用这些原则创建 Skill**

## 重要说明

1. **始终获得用户确认** — 在步骤 3（项目选择）和步骤 5（创建前）
2. **优先学习而非包装** — 编码知识，而不只是"运行这个工具"
3. **检查许可证** — 如果项目有限制性许可证需说明
4. **注明来源** — 在生成的 Skill 中包含署名
5. **质量优先于速度** — 花时间真正理解项目

## 这个 Skill 不是

- 不是包安装器
- 不是工具包装器
- 是**从最佳项目学习**并将知识编码为可复用的 Skill
