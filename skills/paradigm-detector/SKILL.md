# Paradigm Detector — 范式跃迁检测器

*识别架构瓶颈，判断重构时机*

---

## 范式思维 vs 系统思维

| 维度 | 系统思维 | 范式思维 |
|------|---------|---------|
| 关注层 | 系统内的变量关系 | 系统存在的本体论假设 |
| 变革方式 | 渐进式改进（Kaizen）| 破坏性重构 |
| 危险区 | 在旧范式内过度优化 | 过早范式跃迁 |

**识别信号**：当领域内最好的专家用同样方法却得到递减收益时，提示需要范式跃迁而非系统调优。

---

## 范式识别框架

### 四类范式

| 类型 | 特征 | 例子 |
|------|------|------|
| **技术范式** | 工具/方法升级 | 单体→微服务→Serverless |
| **产品范式** | 重新定义产品本质 | 功能机→iPhone |
| **组织范式** | 改变组织运作方式 | 瀑布→敏捷→DevOps |
| **商业范式** | 改变价值创造方式 | 资源控制→价值流动 |

### 范式层级

```
第一序改变（系统内）→ 优化现有系统
第二序改变（系统外）→ 改变游戏规则
```

## 跃迁信号检测

### 七大信号

| # | 信号 | 说明 | 检测方法 |
|---|------|------|---------|
| 1 | **收益递减** | 专家用同样方法，产出递减 | 历史性能曲线斜率 |
| 2 | **复杂度爆炸** | 增加资源反而降低性能 | 反直觉的性能指标 |
| 3 | **外部颠覆** | 新范式从边缘崛起 | 竞品/创业公司分析 |
| 4 | **内部阻力** | 改进方案越来越多地被拒绝 | 决策日志分析 |
| 5 | **元问题重现** | 同样的根本问题反复出现 | 问题模式识别 |
| 6 | **范式参照** | 其他领域已发生跃迁 | 跨领域类比 |
| 7 | **假设失效** | 过去有效的假设不再成立 | 环境扫描 |

### 信号强度评估

```
信号综合分 = Σ(信号强度 × 权重)

阈值：
- < 30：系统思维优化
- 30-60：准备重构
- > 60：范式跃迁窗口
```

## 重构时机判断

### 重构 vs 跃迁

| 决策 | 条件 |
|------|------|
| **继续系统优化** | 收益递减<20%，复杂度可控，外部无颠覆威胁 |
| **准备重构** | 收益递减>30%，复杂度指数增长 |
| **范式跃迁** | 核心假设失效，外部出现新范式，内部阻力达到峰值 |

### 跃迁成本估算

```
跃迁成本 = 迁移成本 + 学习成本 + 机会成本

迁移成本：重写/重构的工程量
学习成本：团队掌握新范式的时间
机会成本：跃迁期间的业务损失

判断：跃迁收益 > 跃迁成本 → 执行
```

## 架构瓶颈识别

### 七类典型瓶颈

| 瓶颈类型 | 表现 | 检测指标 |
|---------|------|---------|
| **技术债型** | 债息侵蚀生产力 | 还债时间 > 新功能时间 |
| **复杂度型** | 调用链过长 | 依赖深度 > 7层 |
| **沟通型** | 团队协调成本 > 执行成本 | 会议时间 > 编码时间 |
| **决策型** | 决策延迟 > 执行时间 | 审批链路 > 5层 |
| **规模型** | 无法线性扩展 | 增加资源不线性提升性能 |
| **创新型** | 渐进改进无法突破 | 竞品差异化扩大 |
| **范式型** | 核心假设不再成立 | 专家收益递减 |

## 使用工具

### 1. 范式扫描

```python
from paradigm_detector import scan_paradigm

result = scan_paradigm(
    system="OpenClaw架构",
    historical_metrics=[...],
    external_signals=[...]
)
# 输出：{paradigm_score, signals, recommendation}
```

### 2. 跃迁时机判断

```python
from paradigm_detector import assess_transition_timing

result = assess_transition_timing(
    current_state="微服务架构",
    metrics={...},
    migration_cost=100,
    opportunity_cost=20
)
# 输出：{should_transition, confidence, next_steps}
```

### 3. 架构瓶颈诊断

```python
from paradigm_detector import diagnose_bottleneck

result = diagnose_bottleneck(
    system_metrics={...}
)
# 输出：{bottleneck_type, severity, recommendations}
```

## 局限与警示

- **过早跃迁风险**：NoSQL神话教训 — 新范式不等于好范式
- **成本低估**：跃迁成本通常被严重低估
- **范式惯性**：现有利益格局会抵抗范式变化

**使用原则**：系统调优3次仍无效 → 启动范式扫描 → 高置信度才跃迁
