# Skill From Masters

站在巨人的肩膀上创建技能。这个 Skill 帮助用户在创建新 Skill 前发现并整合经过验证的专家方法论。

## 核心哲学

大多数专业领域都有杰出的从业者，他们通过书籍、演讲、访谈和框架总结了方法论。基于这些经过验证的方法论创建的技能，比从零开始创建的要有价值得多。

目标不仅仅是"够用" — 而是要达到该领域人类专业知识的最高水平。

## 非技术性技能的关键要求

**技术技能有标准答案。** 写代码、调试、配置系统 — 这些都有相对客观的质量标准。

**非技术性技能的质量差异很大。** 涉及决策、沟通、说服、判断的技能，可能从平庸到世界级不等。差异来自于是否整合了深厚的专业知识。

对于非技术性技能（写作、销售、招聘、产品决策等），遵循以下要求：

### 1. **任务定义要狭窄、具体** ⚠️ 关键
- 任务必须**极其具体且定义明确**
- ❌ 不好："写销售邮件"（太宽泛）
- ✅ 好："给企业 CTO 写 B2B 冷启动邮件"
- ✅ 好："给高管利益相关者写项目状态报告邮件"
- 不同场景需要完全不同的技能
- 如果用户的请求太宽泛，通过提问帮助他们缩窄

### 2. **模型选择：需要 Opus 级别** 🎯 必须
- 非技术性技能**必须使用 Opus 级别模型**（如 Claude Opus）
- 不要使用 Sonnet、Haiku 或其他模型
- Opus 具有处理基于判断、微妙任务所需的推理深度
- 对于这些领域，质量差异是实质性的

### 3. **方法论研究：得出清晰可靠的结论** 🔍 必须
- 继续搜索和沟通，直到得出**非常清晰、可靠的结论**
- 不要止于表面研究
- 来源包括：
  - 模型自身的训练知识
  - 网络搜索当前最佳实践
  - 顶级从业者的黄金示例
  - 反例和常见错误
- 持续迭代，直到方法论非常清晰并得到验证

## 工作流程

### 步骤 1: 理解并缩窄技能意图

**对非技术性技能的关键要求：** 确保任务足够狭窄和具体。

大多数用户会从宽泛的请求开始。你的工作是系统性地帮助他们缩窄，直到任务足够具体，方法论和质量标准都明确。

#### 5 层缩窄框架

使用这个系统方法引导用户从宽泛到具体：

---

**第 1 层：领域识别**

识别宽泛任务的核心领域。

**模板问题：**
```
"[宽泛任务] 根据不同领域可能有不同含义。哪个最接近你的需求？
- [领域 A - 描述]
- [领域 B - 描述]
- [领域 C - 描述]
- 其他？"
```

**示例：**
- 用户："我想要一个做决策的技能"
- 你："决策因领域而异。哪个最接近？
  - 产品决策（功能、路线图、优先级）？
  - 招聘决策（候选人评估、岗位定义）？
  - 投资决策（创业、项目、资源分配）？
  - 战略业务决策（市场进入、合作关系）？
  - 其他？"

---

**第 2 层：上下文约束（5W1H）**

一旦知道领域，使用结构化问题缩窄上下文：

**模板问题：**
- **WHO**：谁使用这个技能？（角色、资历、专业水平）
- **WHAT**：具体产出什么输出或决策？
- **WHERE**：什么组织上下文？（创业公司、企业、远程、行业）
- **WHEN**：在什么阶段/时机使用？（启动、危机、季度、日常）
- **WHY**：主要目标/结果是什么？（对齐、批准、行动、学习）
- **HOW**：有什么约束？（时间限制、流程要求、利益相关者）

**示例：**
- 用户："产品决策"
- 你："让我更好地理解上下文：
  - WHO：你是 PM、创始人还是高管做这些决策？
  - WHAT：是选择功能、优先级还是 go/no-go 发布？
  - WHERE：创业公司（资源有限）还是企业（众多利益相关者）？
  - WHEN：每日冲刺规划还是季度路线图决策？
  - WHY：主要目标 — 团队对齐、高管批准还是客户验证？"

---

**第 3 层：比较缩窄**

呈现 2-3 个相似但不同的场景。强制选择或澄清。

**模板：**
```
"在[缩窄的领域]内，我想确保针对正确的场景。哪个最接近？

A. [具体场景 A - 带上下文]
B. [具体场景 B - 带上下文]
C. [具体场景 C - 带上下文]
D. 都不是 / 组合

哪个最匹配？"
```

---

**第 4 层：边界验证（通过否定）**

确认技能**不**包括什么。这澄清了边界。

**模板：**
```
"为了确保对齐，让我确认边界。这个技能：
- ✅ 包括：[具体场景 X]
- ✅ 包括：[具体场景 Y]
- ❌ 不包括：[相关但不同的场景 A]
- ❌ 不包括：[相关但不同的场景 B]
- ❌ 不包括：[相关但不同的场景 C]

这符合你的需求吗？"
```

---

**第 5 层：具体案例锚定**

询问来自他们经验的真实具体例子。这锁定了范围。

**模板：**
```
"你能描述一个你最近需要这个技能的具体情况吗？包括：
- 情况/上下文是什么？
- 你有什么输入？
- 需要产出什么决策/输出？
- 什么让它困难或耗时？
- 理想的结果是什么样的？"
```

---

**停止条件：够狭窄了吗？**

**当所有问题都回答"是"时停止缩窄：**

1. ✅ **独特方法论**：这个特定场景的专家会有独特建议（不是泛泛的）吗？
2. ✅ **明确质量标准**：能判断输出是"优秀"还是"一般"吗？
3. ✅ **具体约束**：有上下文特定的规则、权衡或失败模式吗？
4. ✅ **具体例子**：用户描述了会使用这个的真实场景吗？
5. ✅ **排除替代方案**：清楚这个**不**涵盖哪些相关任务吗？

**如果有任何一个回答"否"，继续缩窄。**

---

### 步骤 2: 识别技能类型

**关键：** 不同技能类型需要根本不同的方法论和质量标准。

查看参考资料 `references/skill-taxonomy.md` 获取完整分类。核心类型包括：

| 类型 | 核心操作 | 关键问题 |
|------|----------|----------|
| **总结** | 压缩 | 需要全面覆盖？ |
| **洞察** | 提取 | 需要找到真正重要的？ |
| **生成** | 创建 | 需要创建新内容？ |
| **决策** | 选择 | 需要做选择？ |
| **评估** | 判断 | 需要质量判断？ |
| **诊断** | 追溯 | 需要找到根本原因？ |
| **说服** | 桥梁 | 需要改变别人的想法？ |
| **规划** | 分解 | 需要路线图？ |
| **调研** | 发现 | 需要收集知识？ |
| **引导** | 引出 | 需要从他人那里提取信息？ |
| **转化** | 映射 | 需要格式转换？ |

**如何识别：**

问用户："根据你的描述，这听起来像一个 **[类型]** 技能 — 目标是 **[核心操作]**。对吗？"

**常见混淆需要澄清：**
- 总结 vs 洞察："你需要全面覆盖，还是只需要关键信号？"
- 决策 vs 评估："你需要做选择，还是判断质量？"
- 调研 vs 洞察："你需要收集信息，还是理解含义？"

---

### 步骤 3: 识别相关领域

将技能映射到一个或多个方法论领域。单个技能可能跨越多个领域。

示例映射：
- "销售邮件技能" → 销售、写作、说服
- "用户访谈技能" → 用户研究、访谈、产品发现
- "演示技能" → 故事线、视觉设计、说服
- "代码审查技能" → 软件工程、反馈、沟通

---

### 步骤 4: 揭示专家方法论（直到非常清晰）

**目标：** 持续直到你有**非常清晰、可靠的结论**。

**第 1 层：本地数据库**
查阅 `references/methodology-database.md` 获取已知框架。

**第 2 层：网络搜索专家**
搜索网络发现额外的专家和方法论：
- 搜索："[领域] 最佳实践 专家"
- 搜索："[领域] 框架 方法论"
- 搜索："[领域] 大师 从业者"

**第 3 层：深入了解选定的专家**
对于有希望的专家，搜索他们的原始内容：
- 搜索："[专家名] 方法论 访谈"
- 搜索："[专家名] [领域] 演讲"
- 搜索："[专家名] 框架 解释"

获取并阅读原始资料（文章、演讲记录、博客）。

**第 4 层：持续迭代直到清晰** ⚠️
- 不要止于第一个搜索结果
- 如果方法论不清晰或有冲突，深入挖掘
- 寻找：
  - 模型自身的知识（你有大量训练数据）
  - 当前网络最佳实践
  - 从业者的黄金示例
  - 反模式和常见错误
- **继续研究循环**，直到你能自信地说："这是经过验证的方法"

对于每个相关领域，呈现：
- 关键专家及其核心贡献
- 具体框架、原则或流程
- 来源材料（书籍、演讲、访谈）
- 方法论的**置信度**（如果低继续搜索）

---

### 步骤 5: 找到黄金示例

在最终确定方法论选择之前，搜索优秀输出的示例：
- 搜索："最佳 [输出类型] 示例"
- 搜索："[输出类型] 模板 [顶级公司]"
- 搜索："获奖 [输出类型]"

理解卓越什么样有助于定义质量标准。

---

### 步骤 6: 协作选择

向用户展示方法论并讨论：
- 哪些框架与他们的目标产生共鸣？
- 方法论之间有冲突要解决吗？
- 他们应该结合多种方法吗？
- 有什么具体原则他们想强调或排除？

引导用户选择 1-3 个将构成技能基础的主要方法论。

---

### 步骤 7: 提取可操作原则

对于每个选定的方法论，搜索并提炼：

**为什么（核心原则）**
- 搜索："[方法论] 核心原则"
- 搜索："为什么 [方法论] 有效"

**如何（具体流程）**
- 搜索："[方法论] 逐步"
- 搜索："[方法论] 实现指南"

**什么（质量标准）**
- 搜索："[方法论] 检查清单"
- 搜索："[方法论] 评估标准"

**陷阱（常见错误）**
- 搜索："[领域] 常见错误"
- 搜索："[方法论] 陷阱 避免"

获取原始资料以获得准确的措辞和细微差别，而不仅仅是摘要。

---

### 步骤 8: 跨验证

比较多个来源的见解：
- 什么原则一致出现？（高置信度）
- 专家在哪里不同意？（标记给用户）
- 每个方法有什么独特之处？（差异化）

综合一个连贯的框架，吸取每个来源的最佳部分。

---

### 步骤 9: 设计测试场景（生成前）

**关键：** 在生成技能之前，设计全面的测试场景。

与用户合作识别：

**多样化的测试用例：**
- 典型场景（常见情况）
- 边缘情况（不寻常但有效）
- 边界条件（方法论可能失效的地方）
- 失败模式（什么可能出错）

**上下文变化：**
- 不同用户专业水平
- 不同组织上下文（创业公司 vs 企业）
- 不同约束（时间、资源、利益相关者复杂性）
- 文化或行业差异

**质量验证：**
- "优秀"输出长什么样？
- 最常见的错误要避免什么？
- 我们如何知道技能是否有效？

记录这些测试场景 — 它们将在生成后用于验证和迭代。

---

### 步骤 10: 生成技能

方法论确认后，测试场景设计好，**使用 Skill Creator 创建最终技能**。

**如何使用：**
使用 Skill 工具调用 `skill-creator:skill-creator`

这确保：
- 正确的 YAML frontmatter（名称、描述）
- 正确的目录结构
- 生成前的验证
- 命令式写作风格（而非第二人称）

**对于非技术性技能，关键：**
- 在 YAML frontmatter 中添加 `model: opus`
- 确保技能使用 Claude Opus，而非较弱的模型

生成的技能应该：

1. 在注释中注明方法论来源（记录出处）
2. 将专家智慧转化为可操作的指令
3. 包含从找到的黄金示例派生出的具体示例
4. 将质量标准作为明确检查点捕捉
5. 包含来自陷阱研究的"不要这样做"反模式
6. 匹配最佳人类从业者的质量标准
7. **将测试场景作为技能自我验证的一部分包含**

---

### 步骤 11: 测试、审查和迭代

**不要止于第一次生成。** 质量通过迭代产生。

1. **运行测试场景**：将技能应用于步骤 9 中设计的每个案例
2. **评估结果**：根据质量标准比较输出
3. **识别差距**：技能在哪里表现不佳？
4. **改进方法论**：我们需要额外的专家指导吗？
5. **重新生成**：根据学习更新技能
6. **重复**：直到技能持续产生优异结果

让用户参与这个评估 — 他们了解领域，能发现细微差别。

---

## 质量检查清单

最终确定前验证：

**范围和清晰度：**
- [ ] 任务定义是否足够狭窄和具体？（不太宽泛）
- [ ] 有人读技能会确切知道何时使用它吗？

**方法论深度：**
- [ ] 我们是否搜索了本地数据库之外？
- [ ] 我们是否找到了原始来源，而不仅仅是摘要？
- [ ] 我们是否得出了**非常清晰、可靠**的方法论结论？
- [ ] 我们是否找到了输出的黄金示例？
- [ ] 我们是否识别了要避免的常见错误？
- [ ] 我们是否跨多个专家进行了交叉验证？

**生成质量：**
- [ ] 对于非技术性技能：我们是否使用 **Opus 模型** 生成？
- [ ] 技能是否编码了具体、可操作的步骤（不是模糊的原则）？
- [ ] 测试场景是否包含在技能中？

**测试和迭代：**
- [ ] 我们是否事先设计了全面的测试场景？
- [ ] 我们是否在多样化的上下文和边缘情况下测试了技能？
- [ ] 我们是否根据测试结果进行了迭代？
- [ ] 技能是否持续产生优异结果？

## 快速参考：缩窄问题流程

```
宽泛请求
    ↓
第 1 层："哪个领域？" → [选一个]
    ↓
第 2 层："5W1H 上下文？" → [回答约束]
    ↓
第 3 层："哪个具体场景？" → [从 2-3 个选项中选择]
    ↓
第 4 层："排除什么？" → [确认边界]
    ↓
第 5 层："给个具体例子" → [描述具体案例]
    ↓
检查停止条件 → [全部 5 个是？]
    ↓
✅ 够狭窄 → 继续步骤 2
❌ 仍然宽泛 → 继续缩窄
```
