# 代码质量评分标准（7 维度）

## 评分原则

评分基于**证据**，不基于印象。每个维度的评分都应该有对应的代码或数据支撑。
总分 = 7 个维度的平均分，保留一位小数。

---

## 7 维度评分标准

### 1. 架构设计（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 分层清晰，每层职责单一，层间依赖方向一致（高层依赖低层），有明确的扩展点 |
| 7-8  | 分层基本清晰，偶有跨层调用，核心设计决策正确 |
| 5-6  | 分层模糊，存在循环依赖或跨层调用，但整体可理解 |
| 3-4  | 架构混乱，职责不清，难以理解数据流向 |
| 1-2  | 无明显架构，所有逻辑堆在少数几个文件 |

**检测方法：**
```bash
# 循环依赖（需要 madge）
npx madge --circular src/ 2>/dev/null

# 单例数量（过多是耦合信号）
grep -rn "static instance" src/ --include="*.ts" | wc -l
```

---

### 2. 工程细节（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 大量边界情况处理，注释中有"为什么"的解释，明显有踩坑经验沉淀 |
| 7-8  | 主要边界情况有处理，关键逻辑有注释 |
| 5-6  | 基本功能完整，但边界情况处理不足 |
| 3-4  | 只处理了 happy path，边界情况基本没有处理 |
| 1-2  | 代码幼稚，缺乏对生产环境的考虑 |

**检测方法：**
```bash
# 注释中的工程经验（踩坑注释）
grep -rn "// 注意\|// NOTE\|// IMPORTANT\|// 踩坑\|// Workaround\|// Fix:" src/ --include="*.ts" | wc -l
```

---

### 3. 可读性（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 命名自解释，注释解释"为什么"，双语注释（国际化项目加分），无魔法数字 |
| 7-8  | 命名基本清晰，关键逻辑有注释 |
| 5-6  | 命名尚可，但注释不足，需要花时间理解 |
| 3-4  | 命名混乱或过于简短（a、b、tmp），缺少注释 |
| 1-2  | 无法在合理时间内理解代码意图 |

---

### 4. 可维护性（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 文件大小合理（< 300 行），历史债务少，向后兼容代码有明确清理计划 |
| 7-8  | 偶有大文件，历史债务可控 |
| 5-6  | 存在明显的大文件（> 500 行），历史兼容代码积累 |
| 3-4  | 多个超大文件，大量 TODO/FIXME，历史债务严重 |
| 1-2  | 代码难以修改，改一处影响多处 |

**检测方法：**
```bash
# 超大文件
find src -name "*.ts" | xargs wc -l | sort -rn | awk '$1 > 500 {print $1, $2}'

# 历史债务
grep -rn "TODO\|FIXME\|HACK\|legacy\|backward compat\|deprecated" src/ --include="*.ts" | wc -l
```

---

### 5. 健壮性（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 错误处理统一，有超时保护，有重试机制，进程崩溃有恢复 |
| 7-8  | 主要错误路径有处理，关键操作有超时 |
| 5-6  | 错误处理不统一，部分操作缺少超时 |
| 3-4  | 大量静默 catch，缺少超时保护 |
| 1-2  | 基本没有错误处理，任何异常都会导致崩溃 |

**检测方法：**
```bash
# 错误处理质量统计
TOTAL=$(grep -rn "} catch" src/ --include="*.ts" | wc -l)
SILENT=$(grep -rn "} catch {" src/ --include="*.ts" | wc -l)
echo "总 catch: $TOTAL, 静默 catch: $SILENT"
echo "静默比例: $(echo "scale=1; $SILENT * 100 / $TOTAL" | bc)%"

# 超时保护
grep -rn "setTimeout\|Promise.race\|timeout" src/ --include="*.ts" | wc -l
```

**静默 catch 比例评分参考：**
- < 10%：健康（+1 分）
- 10-30%：需关注（不加减分）
- > 30%：技术债警告（-1 分）

---

### 6. 测试覆盖（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | 测试文件数 / 源码文件数 > 50%，核心逻辑有单元测试，有 E2E 测试 |
| 7-8  | 测试比例 30-50%，核心模块有测试 |
| 5-6  | 测试比例 10-30%，测试覆盖不均匀 |
| 3-4  | 测试比例 < 10%，只有少量测试 |
| 1-2  | 基本没有测试 |

**检测方法：**
```bash
TEST=$(find . -name "*.test.ts" -o -name "*.spec.ts" 2>/dev/null | wc -l)
SRC=$(find src -name "*.ts" 2>/dev/null | grep -v "\.test\.\|\.spec\." | wc -l)
echo "测试文件: $TEST, 源码文件: $SRC, 比例: $(echo "scale=1; $TEST * 100 / $SRC" | bc)%"
```

---

### 7. 类型安全（/10）

| 分数 | 标准 |
|------|------|
| 9-10 | TypeScript 严格模式，核心接口类型清晰，无 `any` 滥用 |
| 7-8  | 类型覆盖率高，偶有 `any` |
| 5-6  | 类型覆盖不均匀，关键接口有类型但实现层有 `any` |
| 3-4  | 大量 `any`，类型形同虚设 |
| 1-2  | 无类型或类型完全错误 |

**检测方法：**
```bash
# any 使用数量
grep -rn ": any\|as any\|<any>" src/ --include="*.ts" | wc -l

# 是否开启严格模式
cat tsconfig.json | python3 -c "import json,sys; d=json.load(sys.stdin); print('strict:', d.get('compilerOptions',{}).get('strict', False))"
```

## 问题严重程度分级

**P0 - 稳定性风险**（可能导致生产事故）
- 无超时保护的异步操作（Promise 永久挂起）
- 进程崩溃无恢复机制
- 内存泄漏（事件监听器未清理、Map 无限增长）

**P1 - 可维护性问题**（影响长期迭代速度）
- 单文件超过 800 行（职责过多）
- 大量静默 catch（> 30%）
- 历史兼容代码积累无清理计划

**P2 - 代码规范问题**（影响代码质量但不紧急）
- 流式缓冲策略不统一
- 错误处理策略不统一（有的 warn、有的 throw、有的静默）
- 单例模式滥用（> 10 个）
