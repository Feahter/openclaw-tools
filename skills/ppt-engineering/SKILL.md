# PPT Engineering

> 内容策略与结构设计，让每一页都有存在的理由。

## 核心原则

1. **一页一信息** — 观众同一时刻只能处理一个观点（David JP Phillips）
2. **结论先行** — 标题即核心观点，非描述性标签（Barbara Minto）
3. **信噪比最大化** — 每个元素都必须服务于理解，否则删除（Edward Tufte）
4. **张力驱动** — 在"现状"与"愿景"间交替，制造前进动力（Nancy Duarte）

---

## 执行流程

```
收到请求
  |
  v
1. 需求诊断 ──── 收集关键约束（5 分钟对话）
  |
  v
2. 主线提炼 ──── 一句话 Throughline
  |
  v
3. 叙事选型 ──── 匹配内容类型 -> 选择结构模型
  |
  v
4. 大纲输出 ──── Markdown 结构化大纲 + 页面预算
  |
  v
5. 内容填充 ──── 按单页模板逐页填充
  |
  v
6. 视觉规范 ──── 应用设计系统（见 references/visual-design.md）
  |
  v
7. 质量核查 ──── 对照检查清单
  |
  v
8. 生成输出 ──── 调用 frontend-slides skill 或其他工具
```

## 步骤 1: 需求诊断

收集以下关键信息（不必一次问完，分轮对话）：

| 维度 | 问题 | 影响 |
|------|------|------|
| 时长 | 演讲多长时间？ | 页数预算（约 1 页/分钟） |
| 听众 | 谁在听？技术/业务/混合？ | 术语深度、抽象层级 |
| 目标 | 听完后希望他们做什么？ | 叙事模型选择 |
| 场景 | 现场演讲 / 异步阅读 / 录屏？ | 信息密度、备注需求 |
| 素材 | 已有哪些材料？ | 复用 vs 从零 |

---

## 步骤 2: 主线提炼（Throughline）

> "如果听众只记住一件事，那应该是什么？" — Chris Anderson

用一句话概括整个演示的核心信息。所有后续页面都必须服务于这条主线。

**格式**：`[听众] 应该 [行动/认知改变]，因为 [核心论据]`

**示例**：
- "后端团队应该采用事件驱动架构，因为它能将部署频率提升 10 倍"
- "管理层应该批准 AI 试点项目，因为竞品已在 3 个月内实现 ROI"

---

## 步骤 3: 叙事选型

根据内容类型选择叙事结构。详细框架说明见 `references/methodologies.md`。

### 快速选型指南

| 内容类型 | 特征 | 推荐模型 |
|----------|------|----------|
| **问题解决** | 有明确痛点，需要推动方案落地 | SCQA（Minto） |
| **理念传播** | 传递信念、文化、愿景 | Sparkline（Duarte） |
| **技术方案** | 系统架构、技术全景 | ZUI 缩放 |
| **成果汇报** | 展示已完成的工作和成效 | 金字塔（结论先行） |
| **培训教学** | 教授新知识或技能 | 渐进揭示 |

### 大纲格式

```markdown
Throughline: [一句话主线]
模型: [SCQA / Sparkline / ZUI / 金字塔 / 渐进揭示]
页数预算: [N 页核心 + M 页附录]

- [封面] 标题 + 副标题
- [开场] ...
- [核心1] ...
- [核心2] ...
- [核心N] ...
- [总结] 回扣主线，明确 Call to Action
- [Q&A] 备用页（预设 3 个可能被问到的问题）
```

## 步骤 4-5: 内容填充

### 单页结构模板

```
标题 = 核心观点（非描述性标签）
  例: "事件驱动将部署频率提升 10x" 而非 "架构对比"

视觉锚点（占页面 60%+）
  图表 / 大图 / 关键数字 / 代码片段

支撑点（最多 3 个，每个 3-7 词）
  - 支撑 1
  - 支撑 2
  - 支撑 3

备注区（演讲者笔记，口语化）
```

### 信息密度控制

| 场景 | 字体下限 | 要点上限 | 说明 |
|------|----------|----------|------|
| 现场演讲 | 24pt | 3 点/页 | 3 米外可辨识 |
| 异步阅读 | 16pt | 5 点/页 | 可适当增加密度 |
| 录屏讲解 | 20pt | 4 点/页 | 兼顾屏幕阅读 |

### 数据叙事技巧

详见 `references/content-patterns.md` 中的数据可视化和叙事模式。

核心原则：
1. **先说结论，再展数据** — 不让观众自己猜图表含义
2. **提供对比锚** — "相当于 XX 个..."、"是去年的 N 倍"
3. **高亮关键区域** — 用颜色/注释引导视线到数据重点

---

## 步骤 6: 视觉规范

详细设计系统见 `references/visual-design.md`。

**极简检查清单**：
- [ ] 色彩：1 个主色 + 1 个强调色 + 中性灰阶
- [ ] 字体：单一家族，2 个粗细（标题 Bold / 正文 Regular）
- [ ] 留白：内容占比 < 50%，边距 >= 48px
- [ ] 动画：仅使用"出现"和"平滑移动"
- [ ] 一致性：所有页面遵循同一网格系统

---

## 步骤 7: 质量核查

### 四维检查

| 维度 | 检查项 | 通过标准 |
|------|--------|---------|
| **主线** | Throughline 贯穿 | 遮住所有页面标题，串联起来能讲出完整故事 |
| **单页** | 一页一信息 | 去掉标题后，内容仍支撑同一论点 |
| **过渡** | 页间逻辑 | 每页结尾与下页标题形成因果/递进/转折 |
| **视觉** | 3 米测试 | 投影后 3 米外关键信息可辨识 |

### SUCCESs 粘性检查（Chip & Dan Heath）

- [ ] **Simple** — 核心信息是否足够简单？能否用一句话说清？
- [ ] **Unexpected** — 是否有打破预期的元素？开场是否制造了知识缺口？
- [ ] **Concrete** — 是否用了具体案例/数字/故事而非抽象概念？
- [ ] **Credible** — 数据来源是否可信？是否有权威背书或亲身经历？
- [ ] **Emotional** — 是否触及听众关心的事？（利益/恐惧/成就感）
- [ ] **Story** — 是否有叙事弧线而非信息堆砌？

## 步骤 8: 生成输出

### 工具选择

| 场景 | 推荐工具 | 说明 |
|------|----------|------|
| **HTML 演示** | `frontend-slides` skill | 零依赖、动画丰富、浏览器直接打开 |
| **代码演示** | Slidev / Reveal.js | 适合技术分享，支持代码高亮 |
| **正式场合** | python-pptx 生成 .pptx | 需要 PowerPoint 格式时 |
| **快速原型** | Gamma.app | 快速出初稿，再手动调整 |

生成 HTML 演示文稿时，将完成的大纲、内容、视觉规范传递给 `frontend-slides` skill 执行。

## 示例执行

**输入**: "帮我做一个微服务架构的技术分享，30 分钟，听众是后端工程师"

**输出**:

```
Throughline: 后端团队应该从模块化单体起步而非直接微服务，
            因为过早拆分的代价远大于晚拆分的代价。

模型: SCQA
页数预算: 12 页核心 + 3 页附录

大纲:
- [封面] 微服务：先别急着拆
- [S] 我们都经历过的单体之痛（部署 1 次/周，全员待命）
- [C] 但直接跳到微服务的团队，60% 在 18 个月内回退
- [Q] 如何判断"该拆了"的时机？
- [A1] 信号：识别拆分的 3 个前置条件
- [A2] 策略：Strangler Fig 模式逐步迁移
- [A3] 战术：数据边界划分的 2 个关键决策
- [A4] 案例：我们团队的 6 个月演进时间线
- [结果] 部署频率 1 次/周 -> 10 次/天
- [教训] 3 个踩过的坑和避坑指南
- [总结] 不是"要不要微服务"，是"什么时候准备好了"
- [Q&A] 预设问题：服务间通信选型 / 数据一致性 / 团队拆分

设计规范: 深色主题，#00D4FF 强调色，Inter 字体
风险提示: 第 7 页架构图复杂，建议分步动画揭示
```
