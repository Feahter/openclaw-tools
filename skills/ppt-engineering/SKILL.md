# PPT Engineering - PPT 制作方法论

> 融合工程化思维与极简美学，指导从需求到输出的完整流程。

## 1. 需求分析阶段

### 1.1 关键信息收集

收到请求时，先收集：

| 问题 | 目的 |
|------|------|
| 演讲时长？ | 控制页数 |
| 听众是谁？ | 确定专业度和术语 |
| 核心目标？ | 说服/告知/激励 |
| 现有材料？ | 复用 or 从零 |

### 1.2 内容类型判断

| 类型 | 特征 | 举例 |
|------|------|------|
| **问题解决型** | 痛点→方案 | 技术架构转型、问题排查 |
| **理念传播型** | 信念→行动 | 产品发布、团队文化 |
| **复杂系统型** | 全景→细节 | 系统架构、技术全景 |

---

## 2. 结构选型阶段

### 2.1 叙事模型

```
内容类型判断：
├── 问题解决型 → SCQA模型
│   ├── Situation（共识背景）
│   ├── Complication（冲突/变化）
│   ├── Question（核心疑问）
│   └── Answer（解决方案）
├── 理念传播型 → 黄金圈
│   ├── WHY（信念/动机）
│   ├── HOW（方法/差异）
│   └── WHAT（产品/结果）
└── 复杂系统型 → ZUI缩放
    ├── 全景地图（Zoom Out）
    ├── 模块钻取（Zoom In）
    └── 路径回溯（Zoom Back）
```

### 2.2 输出大纲

用 Markdown 列表输出结构：

```markdown
- [封面] 标题 + 副标题
- [S] 背景共识
- [C] 冲突/挑战
- [Q] 核心问题
- [A-1] 方案1
- [A-2] 方案2
- [总结] 核心结论
- [Q&A] 备用页
```

---

## 3. 内容工程阶段

### 3.1 信息密度控制

**10-20-30 铁律**：
- 10 页核心内容（不含附录）
- 20 分钟演讲时长
- 30 号最小字体（中文 24pt）

### 3.2 单页结构模板

```markdown
# 页面标题 = 核心观点（非描述性）

[视觉锚点：图表/大图/关键数字]

• 支撑点1（3-5词）
• 支撑点2（3-5词）
• 支撑点3（3-5词）

[备注区：口语化讲稿，非书面语]
```

### 3.3 数据叙事三要素

- **时间轴**：展示变化过程
- **对比锚**："相当于XX个..."
- **交互点**：二维码链接详情

## 4. 视觉系统阶段

### 4.1 设计系统规范

```yaml
color_system:
  primary: "#007AFF"  # 品牌主色
  secondary: "#34C759"  # 成功/正向
  tertiary: "#FF9500"  # 警告/强调
  neutral_100: "#000000"  # 正文
  neutral_20: "#F2F2F7"  # 背景

typography:
  heading: "Inter Bold"
  body: "Inter Regular"
  scale: [96, 64, 48, 32, 24, 18]

spacing:
  base_unit: 8px
  margin: 64px  # 极端留白
```

### 4.2 返璞归真检查清单

- [ ] 色彩：是否只用 1 个强调色？背景是否纯白/纯黑？
- [ ] 字体：是否单一家族？是否禁用斜体/阴影/渐变？
- [ ] 留白：内容占比是否 < 50%？
- [ ] 动画：是否仅使用"出现/平滑"？
- [ ] 文字：每行是否 < 10 字？

## 6. 质量核查（AQAL四象限）

| 象限 | 检查项 | 通过标准 |
|------|--------|---------|
| 内容 | 单页单观点 | 去掉标题，内容仍支撑同一论点 |
| 逻辑 | 页间过渡 | 每页结尾与下页标题形成因果/递进 |
| 视觉 | 3米测试 | 投影后3米外关键信息可辨识 |
| 技术 | 故障降级 | 断网/无字体时PDF版仍可展示 |

## 7. 执行流程

当用户请求生成 PPT 时：

```
1. 解析需求
   └── 收集：时长、听众、目标、现有材料
   
2. 结构选型
   └── 判断类型 → 选择模型 → 输出大纲（Markdown）
   
3. 内容填充
   └── 按模板填充每页 → 强制限制3点/页
   
4. 视觉设计
   └── 应用设计系统 → 选择工具链
   
5. 生成输出
   └── Slidev / Python-pptx / Gamma API
   
6. 质量核查
   └── 对照检查清单 → 输出风险提示
```

## 8. 示例对话

**User**: "帮我做一个关于微服务架构的技术分享PPT，30分钟，听众是后端工程师"

**执行**:

```
解析完成：工程模式，SCQA结构
大纲：
- [封面] 从单体到微服务：我们的演进之路
- [S] 3年前单体架构的甜蜜与苦涩（部署频率1周/次）
- [C] 用户增长10倍后的连锁故障
- [Q] 如何在不影响业务的情况下拆分？
- [A-1] 策略：按领域驱动设计划分边界
- [A-2] 战术：数据库解耦与事件溯源
- [A-3] 结果：部署频率提升至1天/10次
- [总结] 微服务不是银弹，而是权衡
- [Q&A] 备用页

设计规范：深色模式，霓虹蓝强调色
风险提示：第5页架构图复杂，建议准备分步动画
```
